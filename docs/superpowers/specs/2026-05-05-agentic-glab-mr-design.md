# Agentic GitLab MR workflow for z

## Problem

`z` is a fast local worktree cockpit: it discovers z-managed git worktrees,
tracks their tmux-backed agents, previews terminal activity, and lets the user
start, attach, stop, and delete agents with very little chrome.

The next workflow gap is GitLab merge requests. Emdash provides a broad PR and
review surface, but its model is too heavy for `z`: persistent task rows,
provider layers, remote SSH projects, and rich desktop panels. `z` should take
the useful idea -- merge-request state belongs next to the worktree/agent -- but
preserve its own philosophy:

- stateless and lightweight
- git, tmux, git worktree config, and `glab` as source of truth
- no local task database
- no hidden durable workflow engine
- minimal UI, only speed/focus

The extra product requirement is that some commands should be agent-backed. A
plain `git rebase` or `glab mr rebase` stops at conflicts; an agent-backed
rebase should hand the engineering intent to the selected agent so it can
resolve conflicts, validate, and report back.

## Decision

Add a local GitLab merge-request layer backed only by `glab`.

The layer is in-memory. It is rebuilt by async discovery and discarded on exit.
It does not write any local MR/task state. The only durable writes remain the
existing z-managed worktree metadata in git worktree config (`z.agent`,
`z.base`) and normal git/GitLab state changed by explicit user actions.

### Source Of Truth

- Worktrees: `git worktree list --porcelain`
- Agent metadata: worktree-scoped git config
- Sessions: tmux
- MR state and mutations: `glab`
- Branch state: git commands run in the repo/worktree

No SQLite, JSON state file, background daemon, or long-lived workflow registry.

## User Model

Each visible row is still an agent/worktree row. Merge-request status is compact
metadata on that row, not a new task object.

The user thinks:

1. Pick an agent/worktree.
2. See whether its branch has an MR.
3. Hit the next useful key.
4. If the action needs engineering judgment, `z` assigns that intent to the
   selected agent in that worktree.

## Direct Vs Agent-Backed Actions

### Direct Actions

Use direct `glab` or git commands when the action is bounded and deterministic:

- refresh MR state
- create MR
- view MR details
- open MR in browser
- add a comment
- push branch before MR creation
- merge when GitLab reports the MR is clean/ready

Direct failures surface as one-line status messages.

### Agent-Backed Intent Actions

Use the selected configured agent when the action may require judgment,
multi-step repair, or conflict resolution:

- `rebase`: update the branch against the MR target branch, resolve conflicts,
  run relevant validation, and stop before merging
- `make-ready`: make the MR merge-ready when it is behind, conflicted, blocked,
  failing checks, or has unresolved review feedback
- `review-fix`: read unresolved MR discussions and address them in the worktree

These are not generic wrappers around every command. They are explicit
engineering intents.

## Agent Execution Policy

Do not create a separate maintenance-agent concept. The selected worktree
already has an agent identity from `z.agent`; reuse that configured agent.

When an agent-backed command is invoked:

- If the selected agent is stopped, create the normal z tmux session in that
  worktree and start the configured fresh command with the deterministic intent
  prompt. Use the fresh path, not the resume path, because the intent prompt is
  the command's contract.
- If the selected agent is creating, refuse the action with a one-line status
  message.
- If the selected agent is running, do not silently inject a
  prompt. Surface a status message instructing the user to attach or stop the
  session first. This preserves trust: `z` never types into an unknown live
  shell state.

This policy keeps `z` predictable and avoids hidden orchestration.

## Intent Prompts

Prompts are generated by a small prompt builder. They should be precise,
repeatable, and free of UI wording.

### Rebase Prompt

```text
Rebase this worktree's branch onto <target-branch>.

Requirements:
- fetch the latest refs first
- perform the rebase in this worktree
- resolve conflicts while preserving the intended changes on this branch
- run the relevant focused validation for the files you touched
- do not merge the merge request
- stop with a concise summary of what changed and what validation ran
```

### Make-Ready Prompt

```text
Make this GitLab merge request ready to merge: <mr-url>.

Requirements:
- inspect MR status, CI/check failures, branch-behind state, conflicts, and
  unresolved review feedback
- update this worktree branch as needed
- fix issues required for merge readiness
- run relevant validation
- push the branch when changes are ready
- do not merge the merge request
- stop with a concise summary and any remaining blocker
```

### Review-Fix Prompt

```text
Address unresolved review feedback on this GitLab merge request: <mr-url>.

Requirements:
- inspect unresolved discussions/comments
- make the requested code changes in this worktree
- run relevant validation
- push the branch when changes are ready
- do not merge the merge request
- stop with a concise summary of addressed feedback and remaining threads
```

## UI Design

Keep the current layout: terminal preview above, compact agent table below,
single-line status/footer.

### Agent Table

Add a compact MR cell after the status glyph and before branch, or fold it into
the branch cell if width is tight.

Suggested glyphs:

- blank: no MR discovered
- `!`: MR status unknown or glab failed
- `D`: draft
- `R`: open and review-ready
- `B`: blocked/conflicted/behind/failing
- `✓`: merged

MR glyphs use the existing status color semantics where possible:

- ready/open: `OK`
- blocked/failing/unknown: `FAIL`
- draft/behind/running sync: `BUSY` or `DIM` depending on confidence
- no MR: empty/DIM

Do not add a new accent color.

### Preview

The default preview remains the tmux pane tail.

Add an MR preview mode for the selected row, toggled with `tab` in normal mode.
The MR preview shows only high-signal fields:

- MR identifier and title
- source -> target
- draft/open/merged state
- merge status / detailed merge status when available
- latest pipeline/check summary when available
- unresolved discussions count when available
- URL

If no MR exists, the MR preview shows the direct create action hint.

### Footer

Footer hints are state-sensitive and terse.

Examples:

- no MR: `m create MR`
- open ready MR: `m view`, `M merge`, `o open`, `r rebase`
- blocked/failing MR: `f make-ready`, `r rebase`, `o open`
- draft MR: `f make-ready`, `o open`

Existing normal-mode hints stay available when help is toggled.

## Commands And Keys

Normal mode additions:

- `m`: create MR if none exists; otherwise switch/show MR preview
- `o`: open selected MR in browser
- `r`: agent-backed rebase intent
- `f`: agent-backed make-ready intent
- `v`: agent-backed review-fix intent
- `M`: direct merge only if MR state is clean/ready; otherwise surface
  "not ready; use f make-ready"
- `tab`: toggle selected preview between terminal and MR

Keep destructive/direct MR merge behind a confirmation modal, matching the
current delete confirmation style.

## glab Integration

Add a small module responsible for command construction and output parsing.

Read commands:

- `glab mr list --source-branch <branch> --output json`
- `glab mr view <branch-or-iid> --output json`
- `glab mr view <branch-or-iid> --comments --unresolved --output json` when
  comment/discussion details are needed

Mutation commands:

- create: `glab mr create --fill --source-branch <branch> --target-branch <base> --yes`
- open: `glab mr view <branch-or-iid> --web`
- merge: `glab mr merge <branch-or-iid> --yes`
- comment: `glab mr note <branch-or-iid> --message <message>`

All command builders pass arguments as argv arrays, never shell strings.

Parsing is permissive. GitLab/glab fields can vary by version and instance.
Only a small normalized model is exposed to the app:

```rust
pub struct MergeRequest {
    pub source_branch: String,
    pub target_branch: Option<String>,
    pub iid: Option<u64>,
    pub title: Option<String>,
    pub url: Option<String>,
    pub state: MrState,
    pub draft: Option<bool>,
    pub merge_state: Option<String>,
    pub pipeline_state: Option<String>,
    pub unresolved_count: Option<u32>,
}
```

Unknown or missing fields degrade to `MrState::Unknown` and one-line status
messages, not panics.

## Data Flow

1. Agent discovery finds z-managed worktrees as it does today.
2. MR refresh runs asynchronously after agent refresh, one `glab` lookup per
   visible branch with backpressure.
3. App state stores an in-memory map:

```rust
HashMap<(repo_path, branch), MrSnapshot>
```

4. UI renders the current snapshot.
5. User actions enqueue direct `glab` commands or agent-backed intent commands.
6. After a mutation, refresh the selected branch's MR snapshot.

MR refresh must not block keystrokes or rendering.

## Error Handling

Common failures should produce concise status messages:

- `glab` missing: `glab not found`
- not authenticated: `glab auth required`
- remote unsupported/not GitLab: `no GitLab remote`
- branch has no MR: no status error; render no-MR state
- JSON parse issue: `glab output changed`
- command failed: prefix with operation, e.g. `MR refresh: <stderr tail>`

Never dump raw multi-line errors into the status bar.

## Testing

Unit tests:

- parse `glab mr list --output json` fixtures for no MR, open MR, draft MR,
  merged MR, blocked MR, missing fields, and malformed JSON
- normalize MR state into glyph/status categories
- command builders produce argv vectors with no shell interpolation
- key handling maps new keys only in normal mode
- merge action refuses non-ready MR states
- agent-backed prompt builders include repo/branch/MR/target data and forbid
  merge in rebase/make-ready/review-fix prompts

App-state tests:

- MR snapshots are carried across agent refresh when branch/repo identity still
  matches
- MR refresh failure marks only that branch snapshot as unknown/error
- selected preview toggle does not affect agent discovery or tmux preview state

Integration/manual validation:

- run against a local repo with a GitLab remote and authenticated `glab`
- create MR from a z worktree
- refresh and render MR state
- open MR in browser
- invoke rebase intent on a stopped agent and confirm a tmux session starts in
  the worktree with the generated prompt
- confirm running-agent injection is refused
- merge a clean test MR only after confirmation

## Out Of Scope

- Remote SSH projects
- GitHub PR support
- GitLab issue browsing
- Persistent task/MR database
- Cross-agent planning or agent-to-agent coordination
- Automatically injecting prompts into already-running sessions
- Force merge or merge-anyway flows
- Rich diff viewer, inline comments UI, or full review browser
- CI log scraping beyond status summary available through `glab`

## Anti-Patterns To Avoid

- Turning `z` into a desktop task manager or durable orchestrator
- Adding local workflow state to compensate for data already available from git,
  tmux, or GitLab
- Wrapping every command with an agent just because the mechanism exists
- Hiding deterministic failures that should remain visible
- Typing into a live tmux session without explicit user control
- Adding visual noise: new panels, badges with long labels, or additional accent
  colors
